## 1. 设计原则

### 1.1 权限三层模型（不可妥协的边界划分）

| 层 | 关心什么 | 谁负责 | 数据来源 |
|---|---|---|---|
| **认证 AuthN** | 你是谁 | 网关 `jwt_middleware`（仅验签 + 注入 header） | JWT 自包含 |
| **粗粒度授权 Coarse AuthZ** | 你能进哪个系统/大模块 | 网关 `authz_middleware`（路由级 ACL） | 配置文件 + token 中的 `roles` |
| **细粒度授权 Fine AuthZ** | 你能对这条数据干什么 | **业务系统调 PDP** | PDP 服务 `policy_srv` 实时计算 |

### 1.2 边界铁律

- 网关**不持有**任何领域字段（部门、数据范围、权限码列表、超级管理员白名单）。
- 网关**不直连** identity_srv 或任何业务库。
- 业务领域规则（"医生只能看自己科室的患者"这类）**永远在业务系统调 PDP 时表达**，不进网关、不进 token。
- JWT token **只承载稳定身份信息**，不承载授权决策结果。

## 3. JWT Token Schema

### 3.1 Claims 定义（最终态）

```go
// gateway/internal/application/middleware/jwt_middleware/claims.go
type Claims struct {
    // 标准 OIDC 字段
    Issuer    string `json:"iss"`              // 固定 = identity_srv URL
    Subject   string `json:"sub"`              // user_profile_id (UUID)
    Audience  string `json:"aud"`              // 固定 = "cloudwego-microservice-demo"
    ExpiresAt int64  `json:"exp"`
    IssuedAt  int64  `json:"iat"`
    NotBefore int64  `json:"nbf"`
    JTI       string `json:"jti"`              // Token 唯一 ID（用于黑名单/审计）

    // 业务身份字段（最小集）
    Username  string   `json:"username"`        // 展示用
    Tenant    string   `json:"tenant"`          // = OrganizationID（单值）
    Roles     []string `json:"roles"`           // 角色 code 列表（不是 ID）
}
```

### 3.2 禁止出现的 claim（已删除）

`RoleIDs`、`DepartmentIDs`、`CorePermission`、`DataScope`、`Status`、`OrganizationID`（被 `Tenant` 取代）

### 3.3 设计理由

- **Status 不进 token**：被禁用的用户必须立刻失效，token 缓存的 status 做不到 → 由 PDP 决策时实时查。
- **DepartmentIDs / DataScope 不进 token**：这些是业务规则，下放给 PDP，避免 token 体积膨胀。
- **Roles 用 code 不用 ID**：可读性、跨服务稳定性、ID 变更不影响已签发 token。
- **Tenant 单值**：用户跨组织角色不同时，由 identity_srv 切换组织时重新签发 token（Q2=C 决策）。

## 4. 网关注入的 HTTP Header（业务侧契约）

JWT 验证通过后，网关向下游业务服务注入：

| Header | 来源 | 示例 |
|---|---|---|
| `X-User-Id` | claims.sub | `7a3e...uuid` |
| `X-User-Name` | claims.username | `alice` |
| `X-Tenant-Id` | claims.tenant | `org-uuid` |
| `X-User-Roles` | claims.roles JOIN "," | `doctor,admin` |
| `X-Request-Id` | requestid 中间件 | `req-...` |
| `X-Auth-Token-Jti` | claims.jti（用于审计串联） | `jti-...` |

**业务侧禁止再次解析 JWT**。统一从 header 取，由 SDK 封装（见 §8）。

## 6. 路由级 ACL（YAML 配置驱动）

### 6.1 配置示例

`gateway/config/authz_rules.yaml`：

```yaml
# 公开路径（认证都不需要）
public:
  - GET  /healthz
  - POST /api/v1/auth/login
  - POST /api/v1/auth/refresh
  - GET  /.well-known/openid-configuration
  - GET  /.well-known/jwks.json

# 仅需认证（任何已登录用户）
authenticated:
  - "*  /api/v1/profile/me"
  - "GET /api/v1/menu/mine"

# 角色门禁（路径前缀级别，OR 关系）
roles:
  - prefix: /api/v1/admin/
    require: [admin, superadmin]
  - prefix: /api/v1/identity/
    require: [identity_admin, superadmin]
```

### 6.2 强约束

- **不允许**在 YAML 里写"数据范围"、"部门隔离"、"超级管理员白名单写死的 username"。
- 任何超出"路径前缀 + 角色 OR"的需求，**走 PDP，不写在网关**。
- 规则文件支持热加载（fsnotify），但不作为初版必需特性。

## 7. PDP 服务 `rpc/policy_srv`

### 7.1 为什么自研而不用 OPA

- 复用现有 Casbin 投资（`casbin_middleware/enforcer.go` 共 430 行）；
- 团队 Go 栈，Rego 学习成本不必要；
- 项目规模不需要 OPA 的部署复杂度。

### 7.2 服务结构

```
rpc/policy_srv/
├── biz/
│   ├── handler/         ← Kitex 生成
│   ├── logic/
│   │   ├── enforcer.go      ← 从 casbin_middleware/enforcer.go 迁移并重写
│   │   ├── policy_loader.go ← 从 identity_srv DB 加载策略
│   │   └── decision.go      ← 决策核心
│   └── dal/             ← 与 identity_srv 共库不同 schema/表前缀（Q4=A 决策）
├── kitex_gen/
├── config/
├── docker/
└── wire/
```

### 7.3 RPC 接口设计

`idl/rpc/policy_srv/policy_service.proto`：

```protobuf
syntax = "proto3";
package policy_srv;

service PolicyService {
  // 单点决策
  rpc Check(CheckRequest) returns (CheckResponse);
  // 批量决策（一个请求决多个 (action, resource)）
  rpc BatchCheck(BatchCheckRequest) returns (BatchCheckResponse);
  // 查询主体能做的所有操作（前端菜单/按钮渲染用）
  rpc ListPermissions(ListPermissionsRequest) returns (ListPermissionsResponse);
  // 策略管理（管理后台使用）
  rpc UpsertPolicy(UpsertPolicyRequest) returns (UpsertPolicyResponse);
  rpc DeletePolicy(DeletePolicyRequest) returns (DeletePolicyResponse);
}

message Subject {
  string user_id = 1;
  string tenant = 2;
  repeated string roles = 3;
  // 业务侧可注入的额外属性（部门、岗位等），由业务系统从自己 DB 查后传入
  map<string, string> attributes = 4;
}

message CheckRequest {
  Subject subject = 1;
  string action = 2;            // e.g. "read", "write", "approve"
  string resource = 3;          // e.g. "patient:7a3e..." 或 "patient.*"
  map<string, string> resource_attributes = 4;
}

message CheckResponse {
  bool allowed = 1;
  string reason = 2;            // 拒绝原因（仅 debug，生产可关）
  string data_scope_hint = 3;   // 决策衍生的数据范围提示（可选）
}
```

### 7.4 性能预算

- P99 延迟 < 10ms（同 podman pod 内，无网络跨主机）；
- Enforcer 内存常驻，策略变更通过 etcd watch 触发增量重载；
- 业务侧 SDK 内置 LRU 缓存：`(subject_jti, action, resource) → decision`，TTL = 30s。

## 8. 业务接入 SDK `iamclient`

> **位置（Q5=B 决策）**：顶层独立 module `iamclient/`，加入 `go.work`。不属于 gateway module，避免反向依赖。

### 8.1 API 草案

```go
package iamclient

// Client 是业务系统接入 IAM 的唯一入口
type Client struct { /* ... */ }

func New(cfg Config) (*Client, error)

// FromHTTPHeader 从网关注入的 header 还原 Subject（HTTP 入口）
func (c *Client) FromHTTPHeader(h http.Header) (*Subject, error)

// FromKitexMetadata 从 Kitex metadata 还原 Subject（RPC 入口）
func (c *Client) FromKitexMetadata(ctx context.Context) (*Subject, error)

// Check 单点鉴权（透传到 policy_srv，带本地缓存）
func (s *Subject) Check(ctx context.Context, action, resource string, opts ...CheckOpt) (bool, error)

// MustCheck 鉴权失败直接返回 errors.PermissionDenied
func (s *Subject) MustCheck(ctx context.Context, action, resource string, opts ...CheckOpt) error
```

### 8.2 使用示例

```go
// 业务 handler
func (h *PatientHandler) Get(ctx context.Context, c *app.RequestContext) {
    subject, _ := h.iam.FromHTTPHeader(c.Request.Header)
    if err := subject.MustCheck(ctx, "read", "patient:"+id,
        iamclient.WithResourceAttr("department_id", patient.DepartmentID),
    ); err != nil {
        c.AbortWithError(403, err)
        return
    }
    c.JSON(200, patient)
}
```

### 8.3 接入底线

业务系统**不应出现**：
- `casbin.Enforce(...)` —— 应由 SDK 透传到 PDP；
- `jwt.Parse(...)` —— 应由 SDK 从 header/metadata 解析；
- 直连 `identity_srv` 数据库 —— 应通过 RPC 或 SDK；
- 自行定义"角色码白名单" —— 应通过 PDP 决策。

## 9. identity_srv 改造点

- `identity_srv` 的 RPC 入口接入 `iamclient`，自己也是 PDP 的客户端（吃自家狗粮）。
- `permission` 相关 API 从 `idl/http/permission/`、`gateway/biz/router/permission/` 整体下线，改由 `policy_srv` 暴露管理 API（管理后台调 `policy_srv` 不再调 `identity_srv`）。
- 颁发 JWT 时按 §3.1 schema 写入，不再塞业务字段。
- 暴露 `/.well-known/openid-configuration` + `/.well-known/jwks.json`（让网关代理对外，Q3=B 决策）。

## 10. 决策记录

| # | 问题 | 决策 |
|---|---|---|
| Q1 | Tenant claim 模型 | **A**：`tenant = OrganizationID`（单值）。多组织成员属于后续演进议题，本期不支持。 |
| Q2 | 用户跨组织角色不同 | **C**：切换组织时由 `identity_srv` 重新签发 token，不在 token 中编码 scope。 |
| Q3 | JWKS 出口 | **B**：JWKS 走网关代理路径 `/.well-known/jwks.json`，对外统一入口；`identity_srv` 内部仍是真正的源。 |
| Q4 | policy_srv 数据库 | **A**：演进期与 `identity_srv` 共库不同 schema/表前缀；后期评估拆库（不在本次重构范围）。 |
| Q5 | iamclient SDK 位置 | **B**：顶层独立 module `iamclient/`，加入 `go.work`；不属于 gateway module，避免反向依赖。 |
| Q6 | 密钥算法 | **A**：完全切 RS256；本地开发使用与生产相同形态的 RSA key（gitignore 真密钥，提供 example）。 |
| Q7 | ListPermissions | **A**：必须实现，前端菜单/按钮渲染依赖此接口。 |

## 11. 与 v1 模型的关键差异

| 维度 | v1（旧） | v2（目标态） |
|---|---|---|
| **JWT Claims** | 含 `RoleIDs / DepartmentIDs / OrganizationID / CorePermission / DataScope / Status` | 仅 `sub / username / tenant / roles` + OIDC 标准字段 |
| **签名算法** | HS256 共享密钥 | RS256（identity_srv 签发，网关验签） |
| **网关状态** | 持有 Casbin 策略缓存 + 同步 identity_srv DB | 无状态（仅 Redis 用于 session） |
| **网关中间件** | OTel→ReqID→...→JWT→Casbin→Audit→ETag | trace→cors→error→response→jwt→authz→access_log |
| **细粒度授权位置** | 网关 `casbin_middleware` 中间件 | 业务系统调 PDP 服务 |
| **权限 API 暴露** | `gateway/biz/router/permission/` + `idl/http/permission/` | 由 `policy_srv` 暴露管理 API（管理后台直连） |
| **业务接入** | 自行 `casbin.Enforce` 或调网关代理 | 通过 `iamclient` SDK 调 `policy_srv` |
| **数据范围** | token 携带 `DataScope` 字段 | PDP 决策时返回 `data_scope_hint`，业务侧自行消费 |
