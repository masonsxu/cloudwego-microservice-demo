## 1. TL;DR

四件事，按顺序做：

1. **JWT Token Schema 最小化**：token 只承载稳定身份（`sub / username / tenant / roles`），删除 `RoleIDs / DepartmentIDs / Permission / DataScope / OrganizationID` 等业务字段。
2. **拆出独立 PDP 服务 `rpc/policy_srv`**：把现有 `gateway/.../casbin_middleware/` 整体迁移过去并重写为 RPC 服务；网关变回无状态。
3. **网关中间件瘦身**：网关只保留 认证 + 路由级粗粒度 ACL + access log + 横切关注点。删除 `casbin_middleware/`、删除 `audit_middleware/`。
4. **新增业务接入 SDK `gateway/pkg/iamclient`**（或独立 module）：业务系统读 header + 调 PDP 的最小封装；`identity_srv` 自己也要迁过去吃自家狗粮。

## 2. 第一性：边界划分（不可妥协）

| 层 | 关心什么 | 谁负责 | 数据来源 |
|---|---|---|---|
| **认证 AuthN** | 你是谁 | 网关 `jwt_middleware`（只验签 + 注入 header） | JWT 自包含 |
| **粗粒度授权 Coarse AuthZ** | 你能进哪个系统/大模块 | 网关 `gateway_authz_middleware`（路由级 ACL） | 配置文件 + token 中的 `roles` |
| **细粒度授权 Fine AuthZ** | 你能对这条数据干什么 | **业务系统调 PDP** | PDP 服务实时计算 |

**铁律**：

- 网关**不持有**任何领域字段（部门、数据范围、权限码列表）。
- 网关**不直连** identity_srv 数据库。
- 业务领域规则（"医生只能看自己科室的患者"这类）**永远在业务系统调 PDP 时表达**，不进网关、不进 token。

## 5. 模块详细设计

### 5.1 JWT Token Schema（最终态）

**Claims 定义（仅这些，不可扩展）**：

```go
// gateway/internal/application/middleware/jwt_middleware/claims.go (重写)
type Claims struct {
    // 标准 OIDC 字段
    Issuer    string `json:"iss"`              // 固定 = identity_srv URL
    Subject   string `json:"sub"`              // user_profile_id (UUID)
    Audience  string `json:"aud"`              // 固定 = "cloudwego-microservice-demo"
    ExpiresAt int64  `json:"exp"`
    IssuedAt  int64  `json:"iat"`
    NotBefore int64  `json:"nbf"`
    JTI       string `json:"jti"`              // Token 唯一 ID（用于黑名单/审计）

    // 业务身份字段（最小集）
    Username  string   `json:"username"`        // 展示用
    Tenant    string   `json:"tenant"`          // = OrganizationID（见待决策 Q1）
    Roles     []string `json:"roles"`           // 角色 code 列表，**不是 ID**
}
```

**禁止出现**的 claim（删除清单）：
- `RoleIDs`、`DepartmentIDs`、`CorePermission`、`DataScope`、`Status`

**理由**：
- `Status` 应在 PDP 决策时实时查（被禁用的用户必须立刻失效，token 缓存的 status 做不到）
- `DepartmentIDs`、`DataScope` 是业务规则，下放给 PDP
- `Roles` 用 code 不用 ID：可读性 + 跨服务稳定性 + ID 变更不影响已签发 token

### 5.2 网关注入的 HTTP Header（业务侧契约）

JWT 验证通过后，网关注入下游：

| Header | 来源 | 示例 |
|---|---|---|
| `X-User-Id` | claims.sub | `7a3e...uuid` |
| `X-User-Name` | claims.username | `alice` |
| `X-Tenant-Id` | claims.tenant | `org-uuid` |
| `X-User-Roles` | claims.roles JOIN "," | `doctor,admin` |
| `X-Request-Id` | 已有 requestid 中间件 | `req-...` |
| `X-Auth-Token-Jti` | claims.jti（用于审计串联） | `jti-...` |

**业务侧禁止再次解析 JWT**。统一从 header 取，由 SDK 封装（见 5.5）。

### 5.4 路由级 ACL 规则（YAML 配置驱动）

**示例**：`gateway/config/authz_rules.yaml`

```yaml
# 公开路径（认证都不需要）
public:
  - GET  /healthz
  - POST /api/v1/auth/login
  - POST /api/v1/auth/refresh
  - GET  /.well-known/openid-configuration
  - GET  /.well-known/jwks.json

# 仅需认证（任何已登录用户）
authenticated:
  - "*  /api/v1/profile/me"
  - "GET /api/v1/menu/mine"

# 角色门禁（路径前缀级别，OR 关系）
roles:
  - prefix: /api/v1/admin/
    require: [admin, superadmin]
  - prefix: /api/v1/identity/
    require: [identity_admin, superadmin]
```

**强约束**：
- 不允许在 YAML 里写"数据范围"、"部门隔离"、"超级管理员白名单写死的 username"。
- 任何超出"路径前缀 + 角色 OR"的需求，**走 PDP，不写在网关**。
- 规则文件支持热加载（fsnotify），但不强制 Phase 1 实现。

### 5.5 PDP 服务 `rpc/policy_srv`（新增）

**为什么自研而不用 OPA**：复用现有 Casbin 投资（`casbin_middleware/enforcer.go` 430 行）；团队 Go 栈、Rego 学习成本不必要；项目规模不需要 OPA 的部署复杂度。

**目录骨架**：

```
rpc/policy_srv/
├── biz/
│   ├── handler/         ← Kitex 生成
│   ├── logic/
│   │   ├── enforcer.go      ← 从 casbin_middleware/enforcer.go 迁移并重写
│   │   ├── policy_loader.go ← 从 identity_srv DB 加载策略（独立连接）
│   │   └── decision.go      ← 决策核心
│   └── dal/             ← 直接复用 identity_srv 的 GORM models？见待决策 Q4
├── kitex_gen/
├── config/
├── docker/
└── wire/
```

**RPC 接口（IDL：`idl/rpc/policy_srv/policy_service.proto`）**：

```protobuf
syntax = "proto3";
package policy_srv;

service PolicyService {
  // 单点决策
  rpc Check(CheckRequest) returns (CheckResponse);
  // 批量决策（一个请求决多个 (action, resource)，给业务侧批量 ACL 用）
  rpc BatchCheck(BatchCheckRequest) returns (BatchCheckResponse);
  // 查询主体能做的所有操作（前端菜单/按钮渲染用）
  rpc ListPermissions(ListPermissionsRequest) returns (ListPermissionsResponse);
  // 策略管理（管理后台使用，等价于现 identity_srv 的 permission API）
  rpc UpsertPolicy(UpsertPolicyRequest) returns (UpsertPolicyResponse);
  rpc DeletePolicy(DeletePolicyRequest) returns (DeletePolicyResponse);
}

message Subject {
  string user_id = 1;           // sub
  string tenant = 2;            // tenant
  repeated string roles = 3;    // 角色 code
  // 业务侧可注入的额外属性（部门、岗位等），由业务系统从自己 DB 查后传入
  map<string, string> attributes = 4;
}

message CheckRequest {
  Subject subject = 1;
  string action = 2;            // e.g. "read", "write", "approve"
  string resource = 3;          // e.g. "patient:7a3e..." 或 "patient.*"
  // 业务侧的资源属性（owner_id、department_id 等），用于 ABAC 决策
  map<string, string> resource_attributes = 4;
}

message CheckResponse {
  bool allowed = 1;
  string reason = 2;            // 拒绝原因（仅 debug，生产可关）
  // 决策衍生的数据范围提示（可选，业务侧自行决定是否消费）
  string data_scope_hint = 3;
}
```

**性能预算**：
- P99 延迟 < 10ms（同 podman pod 内，无网络跨主机）
- Enforcer 内存常驻，策略变更通过 etcd watch 触发增量重载（不做全量轮询）
- 业务侧 SDK 内置 LRU 缓存：`(subject_jti, action, resource)` → decision，TTL = 30s

### 5.6 业务接入 SDK `gateway/pkg/iamclient`

**API 草案**：

```go
package iamclient

// Client 是业务系统接入 IAM 的唯一入口
type Client struct { /* ... */ }

func New(cfg Config) (*Client, error)

// FromHTTPHeader 从网关注入的 header 还原 Subject（业务系统在请求入口调用）
func (c *Client) FromHTTPHeader(h http.Header) (*Subject, error)

// FromKitexMetadata 从 Kitex metadata 还原 Subject（RPC 入口调用）
func (c *Client) FromKitexMetadata(ctx context.Context) (*Subject, error)

// Check 单点鉴权（透传到 policy_srv，带本地缓存）
func (s *Subject) Check(ctx context.Context, action, resource string, opts ...CheckOpt) (bool, error)

// MustCheck 鉴权失败直接返回 errors.PermissionDenied
func (s *Subject) MustCheck(ctx context.Context, action, resource string, opts ...CheckOpt) error
```

**目标使用样态**：

```go
// 业务 handler
func (h *PatientHandler) Get(ctx context.Context, c *app.RequestContext) {
    subject, _ := h.iam.FromHTTPHeader(c.Request.Header)
    if err := subject.MustCheck(ctx, "read", "patient:"+id,
        iamclient.WithResourceAttr("department_id", patient.DepartmentID),
    ); err != nil {
        c.AbortWithError(403, err)
        return
    }
    c.JSON(200, patient)
}
```

**底线**：业务系统不应出现 `casbin.Enforce(...)`、`jwt.Parse(...)`、直连 identity_srv 数据库等代码。

### 5.7 identity_srv 改造点（吃自己狗粮）

- `identity_srv` 的 RPC 入口也接入 `iamclient`，用统一 SDK 鉴权（自己也是 PDP 的客户端）。
- `permission` 相关 API 从 `idl/http/permission/`、`gateway/biz/router/permission/` 整体下线，改由 `policy_srv` 暴露管理 API（管理后台调 `policy_srv` 不再调 identity_srv）。
- 颁发 JWT 时按 5.1 schema 写入，不再塞业务字段。
- 暴露 `/.well-known/openid-configuration` + `/.well-known/jwks.json`（让网关和未来外部业务系统都能拉公钥）。

## 8. 反目标（明确不做）

- ❌ 不引入外部 IAM（Keycloak / Zitadel / Casdoor）—— `identity_srv` 已经是自研 IAM，不重复造轮子也不替换。
- ❌ 不替换 Hertz 网关为 Envoy / APISIX / Higress —— 与本项目"CloudWeGo 实践"的整体定位冲突。
- ❌ 不重写 `web/` 前端 —— 仅在权限/菜单 API 契约变化时调整 `web/src/api/` 适配层。
- ❌ 不引入 OPA / Cerbos —— 复用 Casbin。
- ❌ 不动现有可观测性栈（OTel / zerolog） —— 新模块沿用现有约定。
- ❌ 不"顺便"重构 `rpc/identity_srv` 的领域模型 —— 仅做必要适配（删 permission API、接 SDK）。

## 9. 验收标准（重构完成的硬指标）

1. **网关无状态**：`gateway` 进程不持有任何 DB 连接到业务库（仅 Redis 用于 session/缓存允许保留）。
2. **JWT 体积**：典型 token < 600 字节（含签名）。
3. **网关 middleware 总行数**：从当前 4045 行降到 < 1500 行。
4. **业务接入成本**：新业务系统接入仅需 ① import `iamclient` ② 写一个 wire provider ③ 在 handler 调 `subject.MustCheck`。三步内完成。
5. **回归测试**：现有 e2e 测试 100% 通过；不允许"因为重构所以删测试"。
6. **文档同步**：`docs/04-权限管理/`、`DESIGN.md`、`AGENTS.md`、`README.md` 与代码一致。
7. **podman kube play 一键起全栈**（含新增 policy-srv Pod）。

## 10. 风险与回滚

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| Phase 2 policy_srv 性能不达 P99<10ms | 中 | 中 | 业务 SDK 内置缓存兜底；最坏退回到 enforcer 内嵌业务进程的方案 |
| Casbin 模型迁移过程中策略丢失/语义变化 | 中 | 高 | 迁移前 `pg_dump casbin_rule` 表；新旧并行跑一周做 diff |
| JWKS 缓存与密钥轮换边界 case | 低 | 中 | JWKS cache TTL ≤ 1h，强制 kid 匹配，参考 hertz-contrib/jwt fork 经验 |
| Web 前端因 `permission` API 下线短暂崩坏 | 高 | 低 | Phase 3 与 Phase 5 之间留 1 天给前端适配；保留旧路由 410 Gone 引导 |

**回滚预案**：每个 Phase 一个独立 PR、可独立回滚 git revert。Phase 2 之前网关改动可以单独回滚不影响 policy_srv。

## 11. 决策记录

> 已由 masonsxu 拍板，Agent 按以下决策执行。

- **Q1（tenant 模型）= A**：`tenant = OrganizationID`（单值）。多组织成员属于后续演进议题，本期不支持。
- **Q2（跨组织角色）= C**：切换组织时由 `identity_srv` 重新签发 token，不在 token 中编码 scope。
- **Q3（JWKS 出口）= B**：JWKS 走网关代理路径 `/.well-known/jwks.json`，对外统一入口；`identity_srv` 内部仍是真正的源。
- **Q4（policy_srv 数据库）= A**：演进期与 `identity_srv` 共库不同 schema/表前缀；后期评估拆库（不在本次重构范围）。
- **Q5（iamclient SDK 位置）= B**：顶层独立 module `iamclient/`，加入 `go.work`；不属于 gateway module，避免反向依赖。
- **Q6（密钥算法）= A**：完全切 RS256；本地开发使用与生产相同形态的 RSA key（gitignore 真密钥，提供 example）。
- **Q7（ListPermissions）= A**：必须实现，前端菜单/按钮渲染依赖此接口。
